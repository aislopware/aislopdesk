## Flags (load-bearing per CLAUDE.md)

- **Touches WIRE protocol / golden corpus:** **NO.** No wire bytes change. `text:`/`csi:`/`esc:` inject through the existing `PaneSessionHandle.sendBytes([UInt8])` → input-mux path (the same path keystrokes already use); no new wire message, no version constant, no golden key. Do **not** run `golden-check` for behavior — but it must stay green (a no-op for us).
- **Touches iOS (`#if os(iOS)`):** **YES (compile-only risk).** `WorkspaceKeyDispatcher` + `KeyChordNormalizer` are `#if os(macOS)`. The registry / routing / `KeybindingPreferences` / `KeybindGrammar` parse code is platform-neutral and compiles on iOS. The text/csi/esc *injection* store hook is platform-neutral (`sendBytes`). New `KeyChord.Key` cases and any editor `baseKey`/glyph additions must compile on iOS too. **Run `bash scripts/check-ios.sh` after the registry + Key-enum + parse work** (iOS rots silently — the editor view is `#if os(macOS)` but the model/registry it reads is shared).
- **Touches float-math / codec / controller:** **NO.** No `*`+`+`, no FMA, no min/max, no FEC/QP/ABR. SwiftLint's `addingProduct`/`fma` rule is irrelevant here.
- **App-layer crypto:** none added (N/A).
- **Untrusted-UDP validate-then-drop:** N/A for the keyboard path, but the **config-binding parser** (`text:`/`csi:`/`esc:`/`unbind:`/`goto_tab:N`) parses *user-authored config text* — apply the same discipline: validate-then-drop a malformed directive (return `nil`, never trap, never force-unwrap, bound any declared length before allocating). Tested headlessly.

## otty → aislopdesk mapping decisions (where 1:1 fails, the faithful equivalent)

1. **Sequential pane cycle `⌘]`/`⌘[` (ES-E1-2) vs current `⌘]`/`⌘[` = next/prev TAB.** This is a genuine collision with the live registry (`tab.next`/`tab.prev` own `⌘]`/`⌘[`). otty's reference table also lists `⌘⇧]`/`⌘⇧[` = next/prev tab and `⌘]`/`⌘[` = focus next/prev pane. **Decision: reconcile to otty.** Move tab cycling to `⌘⇧]`/`⌘⇧[` (`nextTab`/`prevTab`) and bind `⌘]`/`⌘[` to two NEW actions `cyclePaneNext`/`cyclePanePrev` (sequential focus over the active tab's panes in DFS order). This matches both otty's reference page and stories ES-E1-2/ES-E1-3 (which separately want `⌘⇧PageUp/Down` for scroll — distinct from `⌘⇧[`/`⌘⇧]`). Update the `testDefaultChordsMatchTheDocumentedTable` pins accordingly (those pins currently encode the OLD Muxy parity — they are ours to re-scope; record the re-scope in a code comment).
2. **Split-left/up `⌘⌥D`/`⌘⌥⇧D` (ES-E1-1).** Registry has only `splitRight`(⌘D)/`splitDown`(⌘⇧D). ADD `splitLeft`(⌘⌥D)/`splitUp`(⌘⌥⇧D) actions. The tree op: `openChooserPane(.split(axis:))` only appends to the natural (right/down) side; for left/up we split then swap into the leading position. **Decision:** route `.splitLeft`→`store.openChooserPane(.split(axis: .horizontal))` then `store.swapActivePaneInDirection(.left)` is wrong (chooser isn't materialized synchronously for swap). Instead add a `leading: Bool` to the chooser-split path (see WI-2) — `openChooserPane(.split(axis:leading:))` inserts the new chooser leaf on the leading side. Faithful: new pane is left/up of the source and focused.
3. **Scroll keys (ES-E1-3): `⇧PageUp/Down`, `⇧Home/End`, `⌘PageUp/Down`.** otty's reference uses `⌘PageUp/Down` for page, `⌘Home/End` for buffer ends, `⌘⌥↑/↓` for few-lines. The story asks `⇧PageUp/Down` + `⇧Home/End` (terminal-native shift-scroll) AND `⌘PageUp/Down` (command page-jump). **Decision:** bind both families. `⇧PageUp/Down`→`scrollPageUp`/`scrollPageDown` (`scroll_page_fractional:±0.9` ≈ a page), `⇧Home`/`⇧End`→`scrollToTop`/`scrollToBottom`, `⌘PageUp`/`⌘PageDown`→`commandJumpPrev`/`commandJumpNext` (these reuse the EXISTING `jumpToBlockInActivePane(delta:)` OSC-133 command-jump, NOT scroll — matching "command-jumps accordingly"). `⇧`-prefixed chords are allowed here because the bare key is a non-printable named key (PageUp/Home/End) — they don't steal a terminal *letter*; the §5 "⌘/⌥-prefix" rule is about not stealing printable keys, so add a test-pin EXEMPTION for named-key scroll chords (see Test plan).
4. **Font size `⌘+`/`⌘-`/`⌘0` (ES-E1-4).** otty `⌘=`/`⌘−`/`⌘0`. **Decision:** register `increaseFontSize`(⌘=, "+" is shift-=, accept both via the chord's base key = `=`), `decreaseFontSize`(⌘-), `resetFontSize`(⌘0). Conflict: `⌘0` — currently FREE (select-tab is ⌘1…⌘9, no ⌘0). Good. Route to `performBindingAction("increase_font_size"|"decrease_font_size"|"reset_font_size")` — libghostty changes its render point size and emits SIGWINCH only if the *grid* changes; the story's "without reflowing the PTY grid" is satisfied because font-size in libghostty rescales glyphs within the same pane pixel box (cols/rows unchanged unless the box changes). GUI-verifiable only (hang-safety: no surface in tests).
5. **`reopenClosed`, `composer`, `promptQueue`, `sendToChat`, `openQuickly` (ES-E1-5).** Register as registry rows with unique chords + a `routeTree` case that DELEGATES (a stub that calls the owning epic's seam, or a graceful no-op closure) — E1 only guarantees registration + a routable case, the real behavior lands in E3 (reopen), E11 (openQuickly), E12 (composer/queue), E13 (sendToChat). `commandPalette`/`cheatSheet`/`find` already exist. Chords from otty: `reopenClosed`=⌘⇧T, `openQuickly`=⌘⇧O, `composer`=⌘⇧E, `promptQueue`=⌘⇧M, `sendToChat`=⌘⌃↩. Verify each is collision-free against `allBindings`.
6. **`text:`/`csi:`/`esc:` + `unbind:` (ES-E1-6).** No otty config file exists in aislopdesk yet, so the *parser* is the deliverable (a pure grammar), persisted into `KeybindingPreferences` as a new `textBindings` map (chord→literal bytes) + an `unbinds` set (chords whose default action is suppressed). The dispatcher resolves a `textBinding` BEFORE the action table and injects via `sendBytes`. iOS: parse + inject work (shared); chord *recording* UI stays macOS-only.

## Risks / traps

- **The `testEveryChordIsCommandOrOptionPrefixed` pin WILL fail** on the ⇧-prefixed scroll chords — this is intended; amend it with the named-key exemption, never weaken it for printable keys.
- **`⌘=` base key.** `⌘+` is physically ⌘⇧= on most layouts; the chord's base key is `=` and ⇧ may or may not be present. Bind `increaseFontSize` to base `=` with `[.command]` (NO shift) so both ⌘= and the auto-shifted ⌘+ resolve via the lowercased base — verify `charactersIgnoringModifiers` reports `=` for the ⌘+ press (it does on US layouts). Document the layout caveat.
- **iOS silent rot:** the registry, Key enum, overrides mapKey, and KeybindGrammar all compile into the iOS slice. `swift build` on macOS won't catch an iOS break — `check-ios.sh` is mandatory after WI-1, WI-4, WI-6.
- **No-backcompat schema bump:** bumping `KeybindingPreferences` to v3 makes any v2 blob decode-fail to default (intended, per the no-backcompat directive) — confirm the store's `try? decode ?? .init()` swallows it.
- **Don't make a dead chord:** every routing case must DO something (a real op or a documented graceful fallback like `peekAndReply`'s) — the registry's job is "no dead chords."
- **Hang-safety:** never instantiate a real `GhosttySurface` in a test for the font/scroll hooks — use a recording `TerminalSurfaceActions` fake on a `FakePaneSession`.
