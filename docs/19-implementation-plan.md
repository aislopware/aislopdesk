## Tên sản phẩm & module

Product/codename: **Rwork** (remote workspace — khớp repo `rworkspace`). Module prefix `Rwork`.

```
rworkspace/
├── Package.swift                 # SPM: libs + CLI execs + tests (HEADLESS-buildable & testable)
├── Sources/
│   ├── RworkProtocol/            # wire format: framing, MessageType, seq(int64), Hello/Ack — pure Swift, 0 platform dep
│   ├── RworkTransport/           # NWConnection + TCP_NODELAY, dual data/control channel,
│   │                             #   BackedWriter/BackedReader (ET replay 64MB/4MB gate), reconnect handshake
│   ├── RworkHost/                # macOS: PTY (openpty + posix_spawn createSession), session mgr,
│   │                             #   no-buffer PTY↔transport relay (QoS user-interactive), TIOCSWINSZ
│   ├── RworkClient/              # shared client: connection mgr, reconnect, input encoding
│   └── RworkTerminal/            # TerminalSurface protocol + HeadlessSurface (libghostty impl = GUI app target)
│   ├── rwork-hostd/              # exec: headless host daemon (PTY + transport)  ← buildable tonight
│   └── rwork-client/             # exec: headless CLI test client                ← buildable tonight
├── Tests/{RworkProtocol,RworkTransport,RworkHost,RworkClient}Tests/
├── Apps/                         # Xcode projects (GUI; depend on SPM + libghostty XCFramework) — scaffold
│   ├── HostApp-macOS/  ClientApp-macOS/  ClientApp-iOS/
├── ThirdParty/ghostty/           # build script + External.zig patch → libghostty.xcframework
└── docs/
```

**Why headless-first:** the whole PATH 1 byte pipeline (host PTY ↔ TCP/`TCP_NODELAY` ↔ client, with replay-buffer reconnect) is the de-risked core and is fully buildable + unit/integration-testable with **no GUI, no libghostty**. The renderer (libghostty surface in a Metal view) is intrinsically part of the GUI app target and sits behind `TerminalSurface`; its XCFramework build is the only fragile step (needs Zig) and must never block the core.

- **WF-4b** · ✅ green · **fixed the reconnect resume-rebind race** (the `HandshakeReconnectTests` flake, reproduced ~1/40 on `testReconnectWithReceivedAboveAckedReplaysNothing`). **Root cause (object-identity traced):** a live `HostSessionTransport.sendOutput(seq=N+1)` captured the current `.ready` data channel and suspended inside `NWConnection.send`; a concurrent RETURNING_CLIENT `resume()`→`swapChannels` reassigned `self.dataChannel` to the fresh channel and closed the **old** one in its detached `Task`, cancelling the very channel the in-flight send was writing to → `NWConnection.send` completed with POSIX 89 (`ECANCELED`). The catch gated on `isChannelDown` (a re-read of the old channel's `currentState`), but the send-completion error and the `.cancelled` `stateUpdateHandler` are **two independent NWConnection callbacks**: the error raced ahead of the state notification, so `currentState` still read `.ready` → the error was re-thrown fatally. **Fix (retain-not-throw, no symptom patch):** (1) `NWMessageChannel.send` now classifies a completion error whose `NWError` is `.posix(.ECANCELED)` as the typed `RworkTransportError.notConnected` ("channel gone" contract) instead of `.sendFailed`; (2) `HostSessionTransport.sendOutput`/`sendExit` detect "channel lost" robustly **without** relying on the racy state re-read — the bound channel was swapped out from under the send (`self.dataChannel !== captured`), the error is `.notConnected`, or the re-read state is `.cancelled`/`.failed` — and in all cases **retain-and-return** (bytes already in the `ReplayBuffer`, `highestSentSeq` advances only on a *successful* send and only while the channel is still current, so the resume's replay loop re-sends them on the new channel: no loss, no fatal throw); (3) post-await `self.dataChannel === captured` guard so a stale success on a swapped-out channel cannot block the resume flush. **Toolchain trap caught:** factoring the channel-lost decision into a shared actor-isolated `async` helper awaited from the catch block **miscompiled** (`EXC_BAD_INSTRUCTION` on the cooperative pool, surfacing deterministically under the `RelayOrderingTests` 20k-line burst — bisected with stash + lldb, baseline 5/5 green vs. helper-form 100% crash); the detection is therefore written **inline** at both catch sites (generates correct code).
